# Day13 — Docker Compose 도입
 
## 목표
 
- `docker-compose.yml` 을 작성하고 기존 `docker run` 명령을 compose로 전환한다.
- `docker compose up / down` 흐름을 직접 익힌다.
- compose 파일 포트 오타를 재현하고 복구한다.
 
## 오늘 한 일
 
- 기존 `docker run -d --name test-app -p 8080:80 nginx` 명령을 `docker-compose.yml` 로 전환했다.
- `services`, `image`, `container_name`, `ports`, `restart` 항목을 직접 작성했다.
- `docker compose up -d` 로 컨테이너를 실행하고 `curl -I http://localhost:8080` 으로 정상 응답을 확인했다.
- 이후 포트를 `8080:8080` 으로 의도적으로 잘못 설정하고 `docker compose up -d --force-recreate` 로 재실행했다.
- `curl -I http://localhost:8080` 실패를 확인하고 `docker compose ps` 로 컨테이너는 Up 상태임을 확인했다.
- `docker-compose.yml` 포트를 `8080:80` 으로 수정하고 다시 `docker compose up -d --force-recreate` 로 복구했다.
- INC-008 문서를 작성했다.
 
## 오늘 배운 점
 
- `docker run` 명령의 각 옵션이 `docker-compose.yml` 어느 항목에 대응하는지 이해했다.
  - `-d` → 없어도 됨 (`docker compose up -d` 에서 처리)
  - `--name test-app` → `container_name`
  - `-p 8080:80` → `ports: "8080:80"`
  - `nginx` → `image`
- `docker compose up -d` 와 `docker compose up -d --force-recreate` 의 차이를 확인했다.
  - `docker compose up -d`는 설정이 바뀌었는데도 이미 실행 중인 컨테이너가 있으면 재생성하지 않을 수 있다.
  - `--force-recreate` 는 설정 변경 사항이 반드시 반영되도록 컨테이너를 새로 만든다.
- compose에서 포트 오타 재현은 `docker run` 기반 INC-005와 원인이 동일하다.
  - 하지만 compose 기반에서는 수정→재실행 흐름이 `docker rm -f` 없이 `--force-recreate` 만으로 처리된다.
- `docker compose ps` 에서 Up 상태라도 서비스 접근 가능 여부는 별도로 확인해야 한다는 점을 다시 확인했다.
 
## 결과/증거
 
- `docker-compose.yml` 작성 완료
- `INC-008-compose-port-misconfiguration.md` 작성 완료
- compose up 정상 출력 캡처 → `evidence/day13-docker-compose-up`
- 포트 오류 재현 캡처 → `evidence/day13-docker-compose-fail` 
 
## 막힌 점
 
- compose 파일을 처음 작성할 때 `ports` 항목의 형식(`"8080:80"` 문자열)이 낯설었다.
- `--force-recreate` 를 언제 써야 하는지 처음에는 헷갈렸다. 설정이 바뀌면 컨테이너를 반드시 새로 만들어야 변경사항이 반영된다는 점을 이해했다.
