# Day17 — 파일 권한 보안 실습

## 목표

- nginx 설정 파일 권한을 의도적으로 잘못 설정하여 위험 상태를 재현한다
- chmod 777(과다)과 chmod 600(과소) 두 가지 케이스를 모두 확인한다
- stat 명령으로 숫자 권한과 타임스탬프를 읽는 방법을 익힌다
- 주요 운영 파일 4개의 권한을 일괄 점검한다
- 표준 권한(644)으로 복구하고 INC-009를 작성한다

## 오늘 한 일

- `ls -l`로 nginx 설정 파일의 현재 권한(644)을 확인했다
- `chmod 777`을 적용하여 모든 사용자에게 읽기/쓰기/실행 권한이 열린 상태를 재현했다
- `stat /etc/nginx/sites-available/default`로 숫자 권한(0644)과 Modify/Change
  타임스탬프를 확인했다
- nginx.conf, sites-available/default, sshd_config, authorized_keys 4개 파일의
  권한을 일괄 점검했다
- `chmod 600`을 적용한 뒤 `systemctl reload nginx` 후 `curl -I http://localhost`로
  200 OK가 나오는 것을 확인했다
- `ps aux | grep nginx`로 master process(root)와 worker process(www-data) 구조를
  직접 확인했다
- `chmod 644`로 최종 복구했다

## 오늘 배운 점

### 1. chmod 600인데 왜 reload 후에도 200 OK가 났을까?

nginx는 두 종류의 프로세스가 역할을 나눠서 실행된다.

- **master process** → root 권한으로 실행, 설정 파일 읽기 담당
- **worker process** → www-data 권한으로 실행, 실제 HTTP 요청 처리 담당

`chmod 600`은 소유자(root)만 읽을 수 있는 상태다.
설정 파일을 읽는 건 master(root)이기 때문에 600이어도 reload가 성공했다.
worker(www-data)는 설정 파일을 직접 읽지 않으므로 영향이 없었다.

600이 문제가 되는 건 root 권한이 없는 모니터링 에이전트나 도구가
설정 파일을 읽지 못하게 되는 경우다.

### 2. stat의 Modify vs Change 타임스탬프 차이

파일에는 내용과 메타데이터(권한, 소유자)가 분리되어 있다.

- **Modify** → 파일 내용이 바뀐 시각
- **Change** → 권한/소유자 등 메타데이터가 바뀐 시각

오늘 `chmod`를 실행했을 때 파일 내용은 건드리지 않았으므로 Modify는 그대로였고,
권한(메타데이터)을 바꿨으므로 Change만 오늘 날짜로 갱신되었다.
즉 stat의 Change 타임스탬프로 누가 언제 권한을 변경했는지 추적할 수 있다.

### 3. authorized_keys가 644가 아니라 600인 이유

`authorized_keys`는 SSH 접속을 허용할 키 목록이다.
이 파일을 다른 사용자가 읽거나 쓸 수 있다면 등록된 키를 알아내거나
몰래 키를 추가하는 것이 가능해진다.

그래서 SSH 데몬은 authorized_keys 권한이 600이 아니면
파일을 신뢰할 수 없다고 판단하여 키 인증 자체를 거부한다.
단순 권장사항이 아니라 SSH가 동작하는 조건이다.

### 4. 주요 파일 표준 권한 정리

| 파일 | 표준 권한 | 이유 |
|------|-----------|------|
| `/etc/nginx/nginx.conf` | 644 | root만 수정, 누구나 읽기 가능 |
| `/etc/nginx/sites-available/default` | 644 | root만 수정, 누구나 읽기 가능 |
| `/etc/ssh/sshd_config` | 644 | root만 수정, 누구나 읽기 가능 |
| `~/.ssh/authorized_keys` | 600 | 소유자만 읽기, 타인 읽기 시 SSH 키 인증 거부 |

## 막힌 점

- `chmod 600` 적용 후 reload해도 nginx가 정상 동작해서 처음엔 이해가 안 됐다.
  `sudo systemctl reload nginx`는 root 권한으로 실행되고, 설정 파일을 읽는 것도
  master process(root)이기 때문에 600이어도 읽을 수 있었다.

- Modify와 Change 타임스탬프가 처음엔 헷갈렸다.
  Modify는 파일 내용 변경, Change는 권한/소유자 변경 시각으로 별도로 기록된다.
