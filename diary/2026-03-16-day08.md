# Day8 — Docker 설치와 서비스 중지/복구 실습



## 목표

- Docker를 설치하고 기본 동작을 확인한다.
- Docker 서비스 중지/복구를 직접 재현한다.
- 관련 운영 절차를 runbook과 incident 문서로 남긴다.



## 오늘 한 일

- Ubuntu 서버에 Docker를 설치했다.
- `docker version`, `docker ps` 로 Docker 기본 동작을 확인했다.
- `systemctl status docker` 확인 중, Docker daemon이 socket activation 구조와 연결되어 있음을 확인했다.
- `sudo systemctl stop docker` 실행 시 `docker.socket` 이 여전히 active 상태라는 메시지를 확인했다.
- 이후 `sudo systemctl stop docker.socket` 도 함께 실행하여 Docker를 완전히 중지했다.
- `systemctl is-active docker`, `systemctl is-active docker.socket` 결과가 모두 `inactive` 인 것을 확인했다.
- 그 상태에서 `sudo docker ps` 실행 시 Docker daemon 연결 실패를 확인했다.
- 이후 `sudo systemctl start docker`, `sudo systemctl start docker.socket` 으로 다시 기동했다.
- 복구 후 `systemctl is-active docker`, `systemctl is-active docker.socket` 결과가 모두 `active` 인 것을 확인했고, `sudo docker ps` 도 정상 동작했다.
- `runbook/docker.md` 와 `incidents/INC-005-docker-service-down.md` 작성 내용을 정리했다.



## 오늘 배운 점

- Docker는 `docker.service` 만 있는 것이 아니라 `docker.socket` 과 함께 동작할 수 있다.
- 따라서 Docker 장애를 재현할 때 service만 중지하면 완전한 중단이 아닐 수 있다.
- 장애 재현 문서를 쓸 때도 "왜 stop 했는데 완전히 멈추지 않았는지"를 같이 기록해야 한다.

## 막힌 점

- 처음에는 `docker.service` 만 중지하면 Docker가 완전히 내려간다고 생각했다.
- 하지만 systemd 메시지를 통해 `docker.socket` 이 여전히 활성 상태임을 확인했다.
- 이후 Docker 장애 재현 시에는 service와 socket을 함께 확인해야 한다는 점을 이해했다.
