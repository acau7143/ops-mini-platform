## 오늘 배운 점

### TCP 3-way handshake

TCP 연결은 아래 3단계로 맺어진다.

```
클라이언트 → 서버 : SYN      "나 연결하고 싶어"
서버 → 클라이언트 : SYN-ACK  "알겠어, 나도 준비됐어"
클라이언트 → 서버 : ACK      "확인, 연결 시작하자"
```

연결 종료는 FIN으로 서로 확인하고 끊는다.

```
클라이언트 → 서버 : FIN      "나 이제 다 보냈어"
서버 → 클라이언트 : FIN-ACK  "알겠어, 나도 끝낼게"
```

### tcpdump Flags 읽는 법

| Flag | 의미 |
|------|------|
| `[S]` | SYN — 연결 요청 |
| `[S.]` | SYN-ACK — 연결 수락 |
| `[.]` | ACK — 확인 응답 |
| `[P.]` | PSH-ACK — 데이터 전송 |
| `[F.]` | FIN — 연결 종료 |
| `[R.]` | RST — 연결 거부/강제 종료 |

### 정상 연결 vs 실패 연결

```
정상 (8080): S → S. → . → 데이터(P.) → F.
실패 (9999): S → R.
```

RST는 "그 포트에 아무것도 없어"라는 서버의 응답이다.

### localhost가 IPv4/IPv6 둘 다 시도하는 이유

`/etc/hosts` 파일에 localhost가 두 줄로 매핑되어 있기 때문이다.

```
127.0.0.1 localhost    ← IPv4
::1 ip6-localhost      ← IPv6
```

curl이 `localhost`로 요청하면 운영체제가 이 파일을 보고 둘 다 시도한다.
그래서 tcpdump에 `IP6 ::1` 과 `IP 127.0.0.1` 두 줄이 찍혔다.

### nginx 경유 요청과 직접 요청의 차이

| 항목 | 직접 요청 (`curl localhost:8080`) | nginx 경유 (`curl localhost/app`) |
|------|----------------------------------|----------------------------------|
| HTTP 버전 | HTTP/1.1 | HTTP/1.0 |
| 출발 IP | `::1` + `127.0.0.1` (둘 다) | `127.0.0.1` 만 |
| 이유 | curl이 localhost 이름으로 요청 | nginx가 `proxy_pass http://127.0.0.1:8080` IPv4로 명시 |

nginx가 백엔드로 요청을 전달할 때 기본적으로 HTTP/1.0을 사용한다.
`proxy_pass`에 IP를 직접 명시했기 때문에 IPv4만 나타난다.

## 왜 이 기술/방법을 선택했는가

| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| 패킷 분석 도구 | tcpdump | Wireshark | 서버 CLI 환경에서 GUI 없이 사용 가능, 실무 서버에서 주로 쓰임 |
| 캡처 인터페이스 | `lo` (loopback) | `eth0` | localhost 통신이 loopback을 지나가기 때문 |

## 오늘 처음 안 사실

| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| RST가 뭔지 몰랐음 | 포트가 없을 때 서버가 RST로 거부 응답을 보냄 | 서버 OS가 해당 포트에 리스닝 소켓이 없으면 TCP 스택이 RST를 반환 |
| localhost 요청 시 왜 두 줄이 찍히는지 몰랐음 | `/etc/hosts`에 IPv4/IPv6 둘 다 매핑되어 있어 OS가 둘 다 시도 | curl이 hostname을 resolve할 때 두 주소를 모두 얻어 순서대로 시도 |
| nginx가 HTTP/1.0으로 전달하는지 몰랐음 | nginx는 upstream으로 요청을 넘길 때 기본적으로 HTTP/1.0 사용 | nginx의 proxy 모듈 기본값이 HTTP/1.0 (HTTP/1.1로 변경하려면 설정 추가 필요) |

## 막힌 점

- tcpdump Flags를 처음 봤을 때 `[S]`, `[R.]` 등의 의미를 몰랐다.
- localhost 요청 시 IPv4/IPv6 두 줄이 찍히는 이유를 처음에는 "두 번 시도하도록 설정된 것"으로만 이해했는데, `/etc/hosts` 파일 확인 후 실제 이유를 파악했다.
