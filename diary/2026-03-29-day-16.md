# Day16 — 보안 베이스라인 Runbook 작성

## 목표

- runbook/security-baseline.md 작성
- UFW 비활성화 장애 재현 및 복구


## 오늘 한 일

- 서버의 SSH 설정, UFW 상태, 파일 권한, Nginx 보안 헤더 현황을 직접 확인했다.
- 확인 결과를 바탕으로 runbook/security-baseline.md 를 작성했다.
- sudo ufw disable 로 UFW를 비활성화하여 방화벽 없는 상태를 재현했다.
- sudo ufw enable 로 복구 후 sudo ufw status verbose 로 규칙 복구를 확인했다.


## 오늘 배운 점

- 보안 헤더가 없으면 브라우저가 알아서 동작하고,
  있으면 서버가 브라우저 동작을 명시적으로 제어한다.
  예를 들어 X-Frame-Options 가 없으면 클릭재킹 공격(사용자가 보이지 않는 버튼을 클릭하도록 속이는 공격)에 노출될 수 있다.

- UFW와 nginx는 역할이 다른 계층이다.
  UFW는 패킷이 서버에 도달하기 전 네트워크 계층에서 허용/차단을 결정하고,
  nginx는 패킷이 도달한 이후 애플리케이션 계층에서 HTTP 요청을 처리한다.
  그래서 UFW가 꺼져도 nginx는 정상 동작한다.

- UFW Default: deny (incoming) 은 화이트리스트 방식이다.
  명시적으로 허용한 포트(22, 80)만 열리고 나머지는 전부 차단된다.
  이 설정이 없으면 실수로 띄운 서비스가 외부에 바로 노출될 수 있다.

- sudo ufw status verbose 는 단순 status 와 달리
  기본 정책(Default policy)까지 함께 보여준다.

## 결과/증거

- runbook/security-baseline.md 작성
- UFW inactive 상태 캡처
- UFW enable 후 verbose 상태 캡처

## 막힌 점

- UFW가 꺼져도 curl -I http://localhost 가 200 OK 였는데
  처음에는 이상하다고 생각했지만 UFW와 nginx가 다른 계층이라는 걸
  이해하고 나서 자연스럽게 납득이 됐다.
