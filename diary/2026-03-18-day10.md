# Day10 — Nginx Reverse Proxy와 502 재현

## 목표

Nginx가 Docker 앱을 `/app` 경로로 프록시하도록 연결한다.  
또한 `502 Bad Gateway` 가 왜 발생하는지 직접 재현하고 로그를 통해 원인을 확인한 뒤 복구한다.

## 오늘 한 일

- Docker 앱이 호스트 `8080` 포트에서 정상 응답하는지 먼저 확인했다.
- `/etc/nginx/sites-available/default` 파일에 `/app` reverse proxy 설정을 추가했다.
- `location = /app`, `location /app/` 블록을 사용해 `/app` 및 하위 경로를 처리하도록 구성했다.
- `proxy_pass http://127.0.0.1:8080/;` 설정으로 Nginx가 Docker 앱으로 요청을 전달하도록 했다.
- `sudo nginx -t` 로 설정 문법을 확인했다.
- `sudo systemctl reload nginx` 후 `curl http://localhost/app`, `curl http://<PUBLIC_IP>/app` 로 정상 응답을 검증했다.
- 이후 upstream 포트를 일부러 틀리게 설정해 `502 Bad Gateway` 를 재현했다.
- `tail -n 20 /var/log/nginx/error.log` 로 upstream 연결 실패 메시지를 확인했다.
- 정상 포트로 수정하고 reload하여 `/app` 정상 응답으로 복구를 확인했다.
- incident 문서와 nginx runbook에 오늘 내용을 반영했다.

## 오늘 배운 점

- `/app` reverse proxy는 단순히 경로를 추가하는 것이 아니라, Nginx가 받은 요청을 뒤쪽 앱으로 전달하는 구조라는 점을 이해했다.
- `location = /app` 와 `location /app/` 는 처리 범위가 다르며, 둘 다 넣으면 `/app` 와 하위 경로를 함께 다룰 수 있다.
- `502 Bad Gateway` 는 Nginx 자체 문제라기보다 upstream 앱 연결 실패 때문에 발생하는 경우가 많다.
- `proxy_pass` 의 주소나 포트가 실제 앱 실행 상태와 다르면 바로 502가 발생할 수 있다.
- 장애 발생 시 `error.log` 를 먼저 보는 습관이 중요하다는 점을 다시 확인했다.
- 오늘 처음으로 `upstream` 이라는 용어를 알게 되었다. Nginx에서 upstream은 Nginx가 요청을 대신 전달하는 뒤쪽 서버나 애플리케이션을 뜻한다. 이번 실습에서는 Docker 앱이 upstream 역할을 했고 사용자가 `/app` 으로 요청하면 Nginx가 그 요청을 upstream인 `127.0.0.1:8080` 으로 넘겨주는 구조였다. 즉 upstream은 Nginx 뒤에서 실제 요청을 처리하는 대상이라는 의미로 이해했다.

## 막힌 점

- `systemctl reload nginx` 수행 시 systemd 관련 경고 메시지가 보여 혼동이 있었다.
- 처음에는 `/etc/nginx/sites-available/default` 수정 때문에 뜬 경고로 생각했지만, 실제로는 Nginx 설정 파일과 systemd service unit은 다른 개념이라는 점을 구분할 필요가 있었다.
- reverse proxy 설정에서 `/app` 경로 추가와 upstream 전달 개념을 처음에는 분리해서 이해하지 못했지만 이번 실습으로 흐름이 조금 더 명확해졌다.
