# Day31 — 네트워크 트러블슈팅 실습 1 (ss / netstat)

## 목표

- ss -lntp 로 포트 점유 프로세스를 특정하는 방법을 익힌다
- nginx와 컨테이너가 같은 포트를 쓰려는 포트 충돌 상황을 두 방향으로 재현한다
- 충돌 원인 분석 → 복구 흐름을 incident로 남긴다

## 오늘 한 일

- ss -lntp 로 현재 포트 점유 상태를 확인했다
- 시나리오 1: nginx가 80 점유 중인 상태에서 컨테이너가 80 바인딩 시도
  - 컨테이너가 Created 상태로 멈추고 Up이 되지 않음을 확인했다
  - ss -lntp 에 해당 컨테이너가 나타나지 않음을 확인했다
- 시나리오 2: 컨테이너가 80 점유 중인 상태에서 nginx 시작 시도
  - nginx -t 는 성공했지만 실제 시작에서 bind() failed 에러 발생
  - systemctl status nginx 로 에러 로그를 확인했다
- 두 시나리오 모두 충돌 원인 제거 후 정상 복구를 확인했다

## 오늘 배운 점

- ss -lntp 는 단순 포트 목록이 아니라 점유 프로세스까지 특정할 수 있는
  트러블슈팅 도구다
- docker-proxy 는 호스트 포트와 컨테이너 포트를 연결해주는 중간 프로세스다
  컨테이너가 직접 호스트 포트를 잡는 것이 아니라 docker-proxy가 대신 열어준다
- Created 상태는 생성은 됐지만 시작에 실패한 상태다
  스스로 Up으로 전환되지 않으며 사용자가 명시적으로 개입해야 한다
- nginx -t 는 설정 파일 문법만 검사한다
  포트 점유 여부 같은 외부 상태는 감지하지 못하므로
  nginx -t 성공이 nginx 시작 성공을 보장하지 않는다

## 왜 이 기술/방법을 선택했는가

| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| 포트 점유 확인 | ss -lntp | lsof -i :80 | 프로세스명까지 한 번에 확인 가능 |
| 충돌 재현 방식 | 두 방향 모두 재현 | 한 방향만 | nginx→컨테이너, 컨테이너→nginx 둘 다 incident로 남기기 위해 |

## 오늘 처음 안 사실

| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| ss에 docker-proxy가 보이는 이유 | 컨테이너가 직접 포트를 잡는 게 아니라 docker-proxy가 중간에서 연결해준다 | 컨테이너는 격리된 네트워크 안에 있으므로 호스트 포트는 docker-proxy가 대신 열어준다 |
| nginx -t 성공 후 시작 실패 가능 | nginx -t 는 문법만 검사하고 포트 점유는 확인하지 않는다 | bind()는 실제 시작 단계에서 일어나므로 -t 단계에서는 감지 불가 |

- 시나리오 1: 컨테이너 Created 상태 캡처
- 시나리오 2: systemctl status nginx 에서 bind() failed 로그 캡처
- 복구 후 ss -lntp 정상 상태 캡처
