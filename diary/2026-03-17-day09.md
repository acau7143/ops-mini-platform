# Day9 — 앱 컨테이너 띄우기



## 목표

- 테스트 앱 컨테이너를 표준 이름과 포트로 실행한다.
- 포트 매핑 실수를 재현하고 원인을 분석한다.
- 올바른 매핑으로 복구한 뒤 baseline 점검 범위에 앱 체크를 추가한다.



## 오늘 한 일

오늘은 Docker 기반 테스트 앱 컨테이너를 `test-app` 이름으로 실행하고, 호스트 포트 8080을 표준 포트로 정했다. 먼저 `nginx` 이미지를 사용해 컨테이너를 띄웠고, 포트 매핑을 일부러 `8080:8080` 으로 잘못 설정해 접속 실패를 재현했다.

이후 `docker ps`, `curl -I http://localhost:8080`, `ss -lntp | grep 8080`, `docker logs test-app` 명령으로 상태를 확인했다. 컨테이너 자체는 실행 중이었지만, 실제 서비스 포트와 연결된 포트 매핑이 잘못되었다는 점을 확인했다.

문제를 해결하기 위해 잘못 띄운 컨테이너를 삭제하고, `-p 8080:80` 으로 다시 실행했다. 복구 후 `curl -I http://localhost:8080` 에서 `200 OK` 를 확인했고, `curl http://localhost:8080` 으로 정상 응답도 확인했다.

또한 기존 baseline 점검 범위에 앱 컨테이너 상태와 앱 HTTP 응답 확인 항목을 추가할 수 있도록 정리했다.



## 오늘 배운 점

- 컨테이너가 실행 중이라는 사실만으로 서비스 접근이 보장되지는 않는다.
- 포트 매핑은 `호스트포트:컨테이너포트` 순서로 이해해야 한다.
- 장애 분석 시 `docker ps`, `curl`, `ss -lntp`, `docker logs` 를 같이 보면 원인 파악이 쉬워진다.
- Day10에서 Reverse Proxy를 연결하기 전에 Day9에서 백엔드 앱 접근을 확실히 정리해두는 것이 중요하다.
-  Docker의 포트 매핑은 `호스트포트:컨테이너포트` 순서라는 점을 다시 이해했다.  
- `-p 8080:80` 으로 실행하면 호스트의 8080 포트 요청이 컨테이너 내부 80 포트로 전달되므로 `curl http://localhost:8080` 이 정상 동작한다. 반대로 `-p 8080:8080` 으로 설정하면 컨테이너 내부 8080 포트에 실제 서비스가 없을 경우 접속이 실패할 수 있다는 점을 확인했다.

## 막힌 점

처음에는 컨테이너가 Up 상태이므로 서비스도 바로 접근될 것이라고 생각했지만 포트 매핑이 잘못되면 컨테이너가 정상 실행 중이어도 HTTP 요청이 실패할 수 있다는 점을 직접 확인했다.  
앞으로는 컨테이너 실행 여부와 서비스 접근 가능 여부를 분리해서 점검해야 한다.
