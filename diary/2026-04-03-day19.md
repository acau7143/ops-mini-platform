# Day 19 — security-baseline.sh 작성

## 목표
보안 점검 항목을 자동으로 체크하고 결과를 evidence/ 에 날짜별로 저장하는
스크립트를 작성한다.

## 오늘 한 일
- 스크립트 작성 전 체크 항목 6개의 실제 출력 형태를 먼저 확인했다.
- `healthcheck.sh` 구조를 기반으로 `security-baseline.sh` 를 작성했다.
- 체크 항목 6개를 하나씩 붙이고 실행하면서 동작을 확인했다.
- UFW inactive 상태에서 PASS가 잘못 나오는 버그를 발견하고 수정했다.
- 결과를 터미널 출력과 동시에 `evidence/security-{날짜}/report.txt` 에 저장하도록 구성했다.
- 이상 항목은 마지막에 따로 모아서 출력하도록 했다.

## 오늘 배운 점
- 스크립트를 짜기 전에 각 명령어의 실제 출력 형태를 먼저 확인해야 한다.
  조건을 잘못 짜면 항상 PASS나 항상 FAIL이 나오는 버그가 생긴다.
- `inactive` 안에 `active` 가 포함되어 있어서 `grep -q "active"` 만으로는
  정확하게 구분이 안 된다. `grep -q "Status: active"` 처럼 더 구체적으로
  써야 한다.
- `[ -z "$변수" ]` 는 변수가 비어 있는지 확인하는 조건이다.
  find 결과가 없으면 비어 있으므로 PASS, 있으면 FAIL로 판단할 수 있다.
- `-z` 와 반대는 `-n` 이다. (`-n` 은 비어 있지 않으면 true)
- `2>/dev/null` 은 에러 메시지를 버리는 용도다.
  권한 없는 디렉토리 탐색 시 생기는 불필요한 에러를 숨길 때 쓴다.

## 결과/증거
