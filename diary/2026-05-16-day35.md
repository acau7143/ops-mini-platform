# Week5 — 자동화와 네트워크 이해

## 목표

인프라 설계 근거를 문서화하고, 네트워크 트러블슈팅 실습을 통해
실제 패킷 흐름을 확인한다. 배포 자동화 스크립트를 직접 작성하고
롤백 로직까지 검증한다.

---

## 이번 주에 한 일

### Day29 — Terraform Remote State (S3 Backend)

- S3 버킷(`ops-mini-tfstate`)에 `terraform.tfstate` 저장
- `backend.tf` 작성 후 `terraform init` 으로 S3 백엔드 연결
- 로컬 state 삭제 후 S3에서 자동으로 읽어오는 흐름 확인
- 버킷 이름 오타로 `init` 단계 실패 재현 및 복구

### Day30 — AWS 설계 근거 문서화

- VPC, Security Group, IAM Role 구성을 콘솔에서 직접 확인
- `decisions/0010-aws-network-design.md` 작성
- EC2 1대 구조에서 IAM Role 분리의 한계와 실무 개선 방향 정리
- Docker 컨테이너도 EC2 IAM Role을 공유한다는 점 확인

### Day31 — 네트워크 트러블슈팅 1 (포트 충돌)

- `ss -lntp` 로 포트 점유 프로세스 특정
- 시나리오 1: nginx가 80 점유 중 → 컨테이너 80 바인딩 시도 → Created 상태로 멈춤
- 시나리오 2: 컨테이너가 80 점유 중 → nginx 시작 시도 → `bind() failed`
- `nginx -t` 는 포트 점유를 감지하지 못한다는 점 확인
- INC-014 작성

### Day32 — 네트워크 트러블슈팅 2 (tcpdump)

- `tcpdump` 로 실제 패킷 캡처
- TCP 3-way handshake(`SYN → SYN-ACK → ACK`) 직접 확인
- 존재하지 않는 포트(9999)로 요청 시 RST 패킷 확인
- nginx 경유 요청(`/app`)과 직접 요청(`:8080`) 패킷 흐름 비교

### Day33 — deploy.sh 작성

- `log()`, `healthcheck()`, `rollback()` 함수 직접 작성
- `healthcheck()` 에 `return 1` 누락 버그 발견 및 수정
- `SERVICE_URL` 을 9999로 바꿔 헬스체크 실패 재현
- 롤백 동작 `deploy.log` 로 확인

### Day34 — 배포 실패 자동 롤백 문서화

- INC-015 작성 (배포 실패 + 자동 롤백)
- `runbook/deploy.md` 에 자동 배포 섹션 추가

---

## 이번 주 배운 점

- `ss -lntp` 는 포트 목록만 보는 게 아니라 점유 프로세스까지 특정할 수 있다
- `nginx -t` 는 문법만 검사한다. 포트 점유 여부는 실제 시작 단계에서 확인된다
- RST 패킷은 "그 포트에 아무것도 없어" 라는 응답이다
- 롤백의 핵심은 "수동 복구 불가" 보다 "자동화로 다운타임 단축"이다
- Docker 컨테이너는 EC2 IAM Role을 공유한다. 컨테이너 단위 권한 분리는 ECS/IRSA 수준에서 해야 한다

---

## 이번 주 가장 중요했던 이해

`deploy.sh` 를 직접 짠 것이 이번 주 핵심이다.

단순히 동작하는 스크립트가 아니라:
- 함수 구조로 역할을 분리하고
- 실패를 감지해서 자동으로 이전 상태로 되돌리고
- 모든 과정을 타임스탬프 기반으로 evidence에 저장한다

"배포 자동화를 직접 구현했다" 는 포트폴리오에서 설명할 수 있는 경험이 됐다.

---

## 아쉬운 점

- `deploy.sh` 에 `set -euo pipefail` 이 없어서 `docker compose pull` 실패 시 스크립트가 계속 진행될 수 있다
- `log()` 가 메인 흐름에서 호출되지 않아 배포 시작/종료 로그가 없다
- 두 가지 모두 다음 주 스크립트 고도화에서 개선할 수 있다
