# Day38 — Compose 장애 심화 (depends_on / restart policy)

## 목표

컨테이너 의존성과 재시작 정책이 실제로 어떻게 동작하는지 장애 상황에서 직접 확인한다.

## 오늘 한 일

- `depends_on` 제거 후 `nginx-proxy` 만 단독 기동하여 DNS 조회 실패로 인한 반복 재시작 재현
- `restart: unless-stopped` 설정에서 컨테이너 내부 프로세스 종료 시 자동 재시작 확인
- `restart: no` 설정에서 자동 재시작이 안 되는 것 확인
- `docker-compose.yaml.bak` 으로 원복 후 정상 복구 확인
- INC-016 작성

## 오늘 배운 점

- `depends_on` 은 컨테이너가 **시작되는 것**만 기다린다. 컨테이너 안 서비스가 준비됐는지는 보장하지 않는다.
- Docker Compose 네트워크 안에서 컨테이너끼리 통신할 때 `127.0.0.1` 이 아니라 **서비스 이름**을 써야 한다. Docker가 서비스 이름을 내부 DNS로 자동 해석해준다.
- `proxy_pass http://app/` 에서 `app` 은 IP가 아니라 이름이기 때문에 DNS 조회가 필요하다. `app` 컨테이너가 없으면 DNS 조회 실패 → nginx 시작 실패로 이어진다.
- `docker kill` 은 Docker가 "사용자가 의도적으로 종료한 것"으로 판단해서 `unless-stopped` 여도 재시작하지 않는다.
- 컨테이너 내부 프로세스가 죽으면 Docker는 "예상치 못한 종료"로 판단해서 `unless-stopped` 설정 시 자동 재시작한다.

## 왜 이 기술/방법을 선택했는가

| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| 의존성 설정 | `depends_on` | 없음 | 컨테이너 기동 순서를 명시적으로 보장하기 위해 |
| restart 정책 | `unless-stopped` | `always`, `no` | 예상치 못한 종료 시 자동 복구하되, 사용자가 의도적으로 내린 경우는 재시작 안 하기 위해 |

## 오늘 처음 안 사실

| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| 컨테이너 안에서 `127.0.0.1` 이 호스트랑 같은 줄 알았음 | 컨테이너 내부에서 `127.0.0.1` 은 자기 자신을 가리킴 | 컨테이너는 각자 독립된 네트워크를 가짐 |
| `proxy_pass` 에 이름을 쓰면 그냥 연결되는 줄 알았음 | 이름 → IP 변환을 위해 DNS 조회가 필요함 | 컴퓨터는 이름으로 직접 통신 못하고 IP가 필요함 |
| `docker kill` 이 `unless-stopped` 를 무시하는 줄 몰랐음 | Docker가 `kill` 을 사용자 의도적 종료로 판단함 | `unless-stopped` 는 예상치 못한 종료에만 재시작함 |

## 결과·증거

- `docker ps` 에서 `Restarting (1)` 반복 확인
- `curl: (7) Failed to connect to localhost port 8080` 확인
- `restart: unless-stopped` 자동 재시작 확인
- `restart: no` 에서 `Exited (137)` 상태 유지 확인
- 원복 후 `curl -I http://localhost:8080` → 200 OK 확인

## 막힌 점

- `docker kill` 로 죽이면 `unless-stopped` 여도 재시작이 안 됐다. 처음엔 설정이 잘못된 줄 알았지만, Docker가 `kill` 을 사용자 의도적 종료로 판단한다는 것을 알게 됐다.
- 컨테이너 안에 `kill` 명령어가 없어서 `nginx -s stop` 으로 내부 프로세스를 종료했다.
