# Week1 — 운영 기초 구축과 첫 장애 대응



## 목표
EC2 기반 서버를 준비하고 nginx를 올린 뒤
기본 점검 기준과 장애 재현/복구 흐름을 몸에 익히는 것을 목표로 했다.



## 오늘 한 일
- runbook/index.md의 링크와 목차를 정리했다.
- 고정 validate 8개를 실행해 현재 서버가 정상 상태인지 다시 확인했다.
- 외부(Public IP)에서 HTTP 200 OK가 나는지 최신 상태를 다시 확인했다.



## 이번 주에 한 일 정리

### Day1  

 - EC2 Ubuntu 인스턴스를 생성했다.
 - SSH 접속에 성공했다.
 - `curl -I http://localhost` 로 200 OK를 확인했다.



### Day2  

 - UFW로 80/tcp를 차단하여 외부 접속 실패를 재현했다.
 - 다시 허용하여 복구했다.
 - 첫 incident 문서를 남겼다.



### Day3  

 - healthcheck.sh를 직접 작성했다.
 - nginx active 여부, localhost HTTP 응답, port 80 listening 여부를 자동 점검하는 흐름을 만들었다.
 - 운영 점검 자동화의 첫 조각을 만들었다.


### Day4~5
  
 - nginx 설정 검증 흐름을 다시 확인했다.
 - `nginx -t`의 중요성을 이해했다.
 - SSH 하드닝을 안전하게 적용하려면 반드시 재접속 가능성을 먼저 확인해야 한다는 점을 정리했다.



### Day6  
 - log_snapshot.sh를 작성했다.
 - uptime, df, free, ss, nginx 상태, nginx 로그를 evidence용으로 저장하는 흐름을 만들었다.
 - `/app` 경로 reverse proxy를 추가하는 구조를 이해했다.
 - upstream 포트를 일부러 잘못 지정하면 502 Bad Gateway가 난다는 점을 확인했다.
 - proxy_pass 포트가 맞아도 실제 백엔드가 떠 있지 않으면 여전히 502가 날 수 있다는 점을 배웠다.
 - access.md는 Day별 기록이 아니라 고정 참고 문서로 두고
   Day별 작업은 diary, 절차는 runbook, 장애 기록은 incidents에 남기기로 정리했다.



## 이번 주 배운 점

- 서버 운영은 단순히 설치가 끝이 아니라 같은 기준으로 반복 점검하는 습관이 중요하다.
- 장애는 실제로 재현해봐야 로그를 어디서 보고 무엇을 먼저 확인해야 하는지 감이 잡힌다.
- `curl -I http://localhost` 와 외부 Public IP 확인은 서로 다른 의미를 가진다.
 - localhost는 서버 내부 관점
 - Public IP는 외부 사용자 관점
- `nginx -t` 는 설정 변경 전 거의 필수적인 검증 단계다.
- `healthcheck.sh`, `log_snapshot.sh` 같은 작은 스크립트도 운영 품질을 꽤 높여준다.



## 이번 주 가장 중요했던 이해

이번 주는 "웹서버를 띄웠다"보다
"운영자가 어떤 기준으로 상태를 보고, 장애를 재현하고, 복구하고, 기록하는가"를 처음으로 연습한 주였다.



특히 아래 흐름이 중요했다.
1. 변경한다.  
2. 검증한다.  
3. 일부러 실패시켜 본다.  
4. 복구한다.  
5. 기록한다.  


이 흐름이 앞으로 프로젝트 전체의 기본 패턴이 될 것 같다.



## 아쉬운 점

- 아직 문서 구조가 완전히 안정된 것은 아니다.
- runbook / incident / diary의 경계가 처음에는 조금 헷갈렸다.
- nginx reverse proxy와 backend 연결은 아직 완성 단계가 아니라 이해 단계에 가깝다.
