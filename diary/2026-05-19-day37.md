# Day37 — scripts/ 전체 정리 및 README 작성

## 목표

스크립트 4개가 유기적으로 연결된 구조를 문서로 완성한다.

## 오늘 한 일

- 스크립트 4개(`healthcheck.sh`, `log_snapshot.sh`, `security-baseline.sh`, `deploy.sh`) 코드를 직접 읽고 역할을 정리했다.
- `scripts/README.md` 를 작성했다.
  - 각 스크립트의 목적, 실행 방법, 출력 파일 구조, 연결 Runbook 명시
  - 장애 대응 시 권장 실행 순서 정리
  - 배포 흐름 정리

## 오늘 배운 점

- `healthcheck.sh` 와 `log_snapshot.sh` 는 언뜻 비슷해 보이지만 목적이 다르다.
  - `healthcheck.sh` → "지금 살아있나?" 빠른 상태 확인
  - `log_snapshot.sh` → "왜 이렇게 됐나?" 증거 수집
- 장애 대응 시 `log_snapshot.sh` 를 먼저 실행해야 하는 이유: 장애 상태는 시간이 지나면 복구되거나 덮어씌워질 수 있기 때문에 증거를 먼저 확보해야 한다.
- 스크립트가 있어도 README 없이는 면접관이나 외부인이 구조를 파악할 수 없다. README는 "뭘 했는지"가 아니라 "왜, 어떻게 연결되는지"를 설명하는 문서다.

## 왜 이 기술/방법을 선택했는가

| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| 문서 형식 | Markdown README | Notion 정리 | GitHub에서 바로 보이고 코드와 함께 관리 가능 |
| 실행 순서 명시 | 장애 대응 흐름으로 정리 | 단순 나열 | 스크립트 간 관계와 사용 맥락을 명확히 전달하기 위해 |

## 오늘 처음 안 사실

| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| healthcheck와 log_snapshot이 비슷해 보임 | 실행 목적과 타이밍이 다름 | healthcheck는 현재 상태 판단, log_snapshot은 장애 시점 증거 확보 용도 |
| 장애 대응 순서가 왜 중요한지 몰랐음 | 증거 먼저 확보, 그 다음 분석 | 장애 상태는 복구되면 사라지므로 먼저 저장해야 함 |
