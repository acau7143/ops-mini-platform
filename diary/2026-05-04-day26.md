# Day26 — Terraform apply + IaC Incident

## 목표

- terraform apply 실행 후 리소스 생성 확인
- 오류 재현 및 원인 분석
- terraform destroy로 리소스 정리
- INC-013 작성

## 오늘 한 일

- `terraform plan` 으로 EC2 + Security Group 생성 계획 확인
- `terraform apply` 1차 실행 → IAM Role 권한 부족으로 403 실패
- AWS 콘솔에서 `ops-mini-cloudwatch-role`에 `AmazonEC2FullAccess` 추가
- `terraform apply` 2차 실행 → t2.micro 프리티어 불가로 400 실패
- `main.tf`에서 `instance_type`을 `t3.micro`로 수정
- `terraform apply` 3차 실행 → EC2 + Security Group 생성 성공
- `terraform show`로 리소스 상태 확인
- `terraform destroy`로 리소스 2개 삭제 완료
- INC-013 문서 작성

## 오늘 배운 점

- EC2 위에서 Terraform을 실행하면 AWS 자격증명을 IAM Role에서 가져온다.
  따라서 IAM Role에 필요한 권한이 없으면 Terraform도 AWS 리소스를 만들 수 없다.
- `terraform plan`이 성공해도 apply에서 실패할 수 있다.
  인스턴스 타입, AMI ID 같은 값은 실제 AWS API를 호출하는 apply 단계에서 검증되기 때문이다.
- `terraform destroy`는 Terraform이 만든 리소스를 코드 기준으로 삭제한다.
  콘솔에서 직접 삭제하는 것과 달리 state 파일과 동기화되어 깔끔하게 정리된다.
- IaC도 incident 대상이 된다. 설정 오류, 권한 오류, 인스턴스 타입 오류 모두 배포 실패로 이어질 수 있다.

## 왜 이 기술/방법을 선택했는가

| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| IAM 권한 | AmazonEC2FullAccess | 커스텀 최소 권한 정책 | 실습 편의상 FullAccess 사용. 실무에서는 최소 권한 적용 필요 |
| 인스턴스 타입 | t3.micro | t2.micro | t2.micro가 서울 리전 프리티어 대상 아님 |

## 오늘 처음 안 사실

| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| plan 성공 = apply 성공 아닌가 | plan은 문법/구조 검사, apply는 실제 AWS API 호출 | 인스턴스 타입 같은 값은 API 호출 시점에 검증됨 |
| EC2 위 Terraform 권한 출처 | IAM Role에서 자동으로 가져옴 | EC2 메타데이터 서비스를 통해 Role 자격증명 획득 |
