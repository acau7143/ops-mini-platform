# Day24 — 성능 기준선 측정 및 컨테이너 OOM 재현

## 목표

- ab로 nginx 요청 처리량을 측정하고 기준선 수치를 기록한다.
- app 컨테이너에 메모리 제한을 걸고 OOM 상황을 재현한다.
- runbook/performance-baseline.md와 INC-012를 작성한다.

## 오늘 한 일

- `ab -n 1000 -c 10 http://localhost:8080/` 으로 nginx 요청 처리량을 측정했다.
  - 총 1000개 요청을 동시 10개씩 보내는 방식이다.
  - 동시 10개란 항상 10개의 요청이 동시에 진행 중인 상태를 유지하면서 총 1000개를 채우는 방식이다.
- `docker-compose.yaml` 에 `mem_limit: 50m` 을 추가하고 `--force-recreate` 로 재적용했다.
- `docker inspect app | grep -i memory` 로 52428800(50MB) 제한이 실제로 적용됐는지 확인했다.
  - `docker inspect` 는 컨테이너의 상세 설정 정보를 JSON 형태로 출력하는 명령어다.
- `docker inspect app | grep -i oomkilled` 로 `OOMKilled: true` 를 확인했다.
- `docker-compose.yaml` 에서 `mem_limit: 50m` 을 제거하고 재적용하여 복구했다.
- `docker inspect app | grep -i memory` 에서 `Memory: 0` 으로 제한이 해제됐음을 확인했다.

## 오늘 배운 점

- 오늘 성능 기준선을 측정한 이유는 두 가지다.
  첫째, 지금 정상 상태의 수치를 기록해두면 나중에 설정을 바꿨을 때 성능이 나빠졌는지 비교할 수 있다.
  둘째, 메모리 제한 없이 두면 컨테이너가 호스트 메모리를 무한정 사용할 수 있어 호스트 전체에 영향을 줄 수 있다.

- `docker inspect` 로 컨테이너 설정 정보를 확인할 수 있다.
  `grep -i` 는 대소문자 구분 없이 검색하는 옵션이다.
  `OOMKilled` 처럼 대문자가 섞인 키워드를 찾을 때 유용하다.

- OOM(Out of Memory)은 컨테이너가 설정된 메모리 제한을 초과했을 때 커널이 프로세스를 강제 종료하는 것이다.
  오늘은 `mem_limit: 50m` 상태에서 `apt-get` 이 패키지 목록을 받는 과정에서 50MB를 초과해 `Killed` 가 출력됐다.

- `restart: unless-stopped` 가 설정되어 있으면 OOM으로 컨테이너가 종료돼도 자동으로 재시작된다.
  그래서 `docker compose ps` 만 보면 Up 상태로 보여 OOM이 발생했는지 모를 수 있다.
  `docker inspect` 의 `OOMKilled` 필드를 별도로 확인해야 한다.

- `docker compose ps` 와 `docker ps` 의 차이:
  `docker ps` 는 호스트에서 실행 중인 모든 컨테이너를 보여주고,
  `docker compose ps` 는 현재 compose 파일 기준으로 관리되는 컨테이너만 보여준다.

## 막힌 점

- 컨테이너 내부에 stress가 없어서 `apt-get install` 을 시도했는데,
  설치 과정 자체에서 OOM이 발생했다.
  의도한 방식은 아니었지만 결과적으로 OOM 재현은 성공했다.
- `docker compose ps` 상 Up 상태여서 처음에는 OOM이 발생했는지 몰랐다.
  `docker inspect` 로 별도 확인하는 습관이 필요하다는 점을 이해했다.
