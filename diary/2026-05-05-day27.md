# Day27 — Terraform 변수 구조화

## 목표
- variables.tf: AMI ID, 인스턴스 타입, 키 이름 변수 분리
- outputs.tf: EC2 Public IP 출력
- terraform.tfvars: 환경별 값 관리
- 하드코딩 제거 → 재사용 가능한 구조로 전환

## 오늘 한 일
- main.tf에서 하드코딩된 값 3개(ami_id, instance_type, key_name)를 확인했다
- variables.tf에 변수 3개를 정의했다
- terraform.tfvars에 실제 값을 분리해서 관리하도록 했다
- main.tf에서 하드코딩 값을 var.변수이름 참조로 교체했다
- outputs.tf에 EC2 Public IP 출력을 추가했다
- terraform validate로 문법 오류가 없음을 확인했다
- terraform plan으로 변수값이 tfvars에서 올바르게 읽혀오는 것을 확인했다

## 왜 이 기술/방법을 선택했는가

| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| 변수 관리 | variables.tf + tfvars | 환경변수(TF_VAR_) | 파일로 관리해야 GitHub에 추적 가능 |
| 값 분리 | terraform.tfvars | main.tf 직접 수정 | 값 변경 시 tfvars만 수정하면 전체 반영 |

## 오늘 처음 안 사실

| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| var.이 왜 붙는지 | 변수 참조 문법이 var.변수이름 형식임 | variables.tf에 정의된 변수를 가져올 때 Terraform 고유 문법 |
| (known after apply)가 뭔지 | apply 후 AWS가 채워주는 값 | EC2 생성 전이라 IP가 아직 없는 상태 |
| tfstate가 왜 중요한지 | Terraform이 AWS 현재 상태를 추적하는 파일 | 없으면 기존 리소스를 모르고 중복 생성 시도함 |

## 결과·증거
- terraform validate → Success
- terraform plan → 2 to add, 0 to change, 0 to destroy
- plan 출력에서 변수값이 tfvars에서 정상 읽혀오는 것 확인
- instance_public_ip = (known after apply) 출력 확인
