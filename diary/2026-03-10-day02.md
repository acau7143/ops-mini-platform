# Day2 — 외부 HTTP 확인 및 UFW 차단/복구



## 목표



- EC2 내부(localhost)뿐 아니라 외부(Public IP)에서도 nginx가 정상 응답하는지 확인

- 고정 validate 명령으로 서버 기본 상태 점검

- UFW로 HTTP(80) 차단 상황을 재현하고 복구

- 첫 번째 Incident 작성을 위한 근거 확보



## 오늘 한 일



- 고정 validate 명령을 실행해 서버 상태를 점검했다.

- `systemctl is-active nginx`로 nginx 서비스 상태를 확인했다.

- `curl -I http://localhost`로 EC2 내부에서 HTTP 200 OK 응답을 확인했다.

- `ss -lntp | grep -E ':80|:443'`로 80 포트 리슨 상태를 점검했다.

- 로컬 PC에서 Public IP로 접속해 외부에서도 nginx 응답이 오는지 확인했다.

- UFW 상태를 확인하고, HTTP(80) 차단/복구 테스트를 진행할 준비를 했다.



## 오늘 배운 점



- `localhost`에서 200 OK가 나온다는 것은 nginx 서비스 자체가 서버 내부에서는 정상 동작하고 있다는 뜻이다.

- 외부 접속이 되지 않을 때는 nginx 자체 문제뿐 아니라 Security Group, UFW 같은 네트워크/방화벽 계층도 함께 봐야 한다.

- `systemctl is-active nginx`, `curl -I http://localhost`, `ss -lntp`, `access.log`, `error.log`는 각각 다른 관점에서 상태를 확인하는 명령어다.

- 운영형 프로젝트에서는 “된다/안 된다”만 보는 것이 아니라, 내부 응답과 외부 응답을 구분해서 확인하는 습관이 중요하다.



## 결과/증거



- nginx 서비스 상태 확인

- `HTTP/1.1 200 OK` (localhost)

- 포트 80 LISTEN 상태 확인

- 외부 Public IP 접근 확인 결과

- UFW 상태 확인 결과

- Day2 evidence 캡처 파일 저장
