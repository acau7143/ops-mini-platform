# Day29 — Terraform Remote State (S3 Backend)

## 목표
- S3 버킷에 terraform.tfstate 저장
- backend.tf 작성
- 로컬 state 삭제 후 S3에서 읽어오는 흐름 확인
- state 파일 없는 상태에서 오류 재현 및 복구

## 오늘 한 일
- AWS CLI로 S3 버킷(ops-mini-tfstate) 생성
- backend.tf 작성 후 terraform init으로 S3 백엔드 연결
- terraform state push로 로컬 state를 S3에 마이그레이션
- 로컬 terraform.tfstate 삭제 후 terraform plan 실행
  → S3에서 자동으로 state를 읽어와 No changes 확인
- backend.tf의 bucket 이름을 일부러 틀리게 수정
  → terraform init 단계에서 오류 발생 재현
- 버킷 이름 원복 후 terraform init -reconfigure로 복구

## 오늘 배운 점
- terraform.tfstate는 Terraform이 "현재 AWS에 뭐가 있는지"를
  기록한 파일이다. 이게 없으면 코드와 실제 인프라를 비교할 수 없다.
- remote state를 쓰면 로컬 파일이 없어도 S3에서 자동으로 읽어온다.
- backend 설정이 잘못되면 plan도 아니고 init 단계부터 실패한다.
- IAM 정책 변경 후 반영까지 약간의 딜레이가 있을 수 있다.

## 왜 이 기술/방법을 선택했는가

| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| Remote State 저장소 | S3 | Terraform Cloud | 이미 AWS 사용 중이라 별도 서비스 불필요 |
| State 파일 경로(key) | terraform/terraform.tfstate | 루트에 바로 저장 | 버킷 안에서 경로 구분이 명확해짐 |

## 오늘 처음 안 사실

| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| init이 단순 초기화인 줄 알았다 | init 시점에 백엔드 연결을 시도한다 | backend.tf가 있으면 init할 때 S3 버킷 존재 여부를 확인하기 때문 |
| plan하면 state가 저장되는 줄 알았다 | state는 apply 또는 push할 때 써진다 | plan은 읽기만 하고 인프라를 변경하지 않기 때문 |
