# Day33 — deploy.sh 작성: 자동 배포 및 롤백 로직 구현

## 목표

- `scripts/deploy.sh` 작성
- 배포 흐름 자동화: pull → up → 헬스체크 → 실패 시 롤백
- evidence 디렉토리에 타임스탬프 기반 결과 저장

---

## 오늘 한 일

- `log()`, `healthcheck()`, `rollback()` 함수를 직접 작성했다
- `healthcheck()` 에 `return 1` 누락으로 롤백이 트리거되지 않는 버그를 발견하고 수정했다
- `SERVICE_URL` 을 `localhost:9999` 로 바꿔 헬스체크 실패를 강제로 재현했다
- 롤백 동작을 `deploy.log` 로 확인했다
- 정상 배포 시 `before.txt` / `after.txt` 저장 확인했다

---

## 오늘 배운 점

- 함수를 분리하면 메인 흐름이 읽기 쉬워지고 변경도 편해진다
- 롤백은 "수동 복구가 불가능한 상황"보다 "자동화로 다운타임을 줄이는 것"이 핵심 목적이다
- `--no-pull` 없이 롤백하면 망가진 새 이미지를 또 당겨오게 된다

---

## 왜 이 기술/방법을 선택했는가

| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| 배포 자동화 | shell script | 수동 docker compose 명령 | 재현 가능하고 GitHub에 남길 수 있음 |
| 롤백 방식 | `--no-pull` 로 로컬 캐시 이미지 재기동 | AMI 스냅샷 복구 | 컨테이너 단위 롤백이 더 빠르고 가볍다 |
| 헬스체크 | `curl -sf -o /dev/null` | 상태코드 숫자 비교 | `-f` 옵션으로 성공/실패를 조건문에 바로 연결 가능 |

## 오늘 처음 안 사실

| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| `#!/usr/bin/env bash` 가 뭔지 몰랐음 | shebang — 운영체제에게 이 파일을 bash로 실행하라고 알려주는 것 | `./파일명.sh` 로 실행할 때 앞에 `bash` 를 안 붙이면 운영체제가 실행 방법을 모르기 때문 |
| `/usr/bin/env` 가 왜 필요한지 몰랐음 | bash 위치가 시스템마다 다를 수 있어서 `env` 에게 찾아달라고 위임하는 것 | 어떤 시스템은 bash가 `/usr/local/bin/bash` 에 있을 수 있음 |
| 타임스탬프에 공백 넣으면 안 되는 줄 몰랐음 | 경로에 공백이 있으면 bash가 두 단어로 쪼개서 인식해 디렉토리 생성 실패 | `$OUTDIR` 이 `test dir` 이면 `test` 와 `dir` 로 분리됨 |
| `healthcheck()` 에 `return 1` 이 왜 필요한지 몰랐음 | `while` 루프가 끝난 뒤 명시적으로 실패를 반환해야 롤백이 트리거됨 | `return 1` 없으면 마지막 명령어 결과가 반환값이 되어 성공으로 판단될 수 있음 |
| `curl -o /dev/null` 역할 몰랐음 | 응답 본문을 버려서 터미널에 HTML이 출력되지 않게 함 | `/dev/null` 은 블랙홀 — 거기로 보내면 출력이 사라짐 |
| `--no-pull` 이 롤백에서 왜 필요한지 몰랐음 | 롤백 시 최신 이미지를 또 당겨오면 망가진 버전이 다시 올라오기 때문 | 로컬에 캐시된 이전 이미지를 그대로 써야 "이전 상태로 복구"가 됨 |

- `evidence/YYYYMMDD_HHMMSS-deploy/after.txt` — 배포 후 컨테이너 상태
- `evidence/YYYYMMDD_HHMMSS-deploy/deploy.log` — 롤백 로그 확인

---

## 막힌 점

- `healthcheck()` 에 `return 1` 이 없어서 롤백이 트리거되지 않았다. `while` 루프가 끝나도 명시적으로 실패를 반환하지 않으면 함수가 성공으로 판단될 수 있다는 점을 직접 확인했다.
- 타임스탬프 경로에 공백이 있으면 디렉토리 생성이 실패한다. `_` 로 대체해서 해결했다.
