# Day3 — healthcheck 스크립트 작성과 점검 기준 연결



## 목표

기존에 정리해둔 baseline check 기준을 다시 보면서, 

반복 점검 항목 일부를 셸 스크립트로 자동 확인할 수 있도록 healthcheck를 작성한다.



## 오늘 한 일

- baseline check 전체를 한 번에 자동화하지 않고 판단이 명확한 항목부터 자동화하기로 방향을 잡음

- `healthcheck.sh`를 직접 작성함

- `systemctl is-active nginx` 결과를 변수에 저장해서 조건문으로 비교함

- `curl`로 localhost 응답 상태를 확인하는 방법을 테스트함

- `awk`를 사용해 HTTP 응답 헤더에서 필요한 값을 추출해봄

- `sh script.sh`와 `./script.sh` 실행 차이, `bash` 문법 차이를 확인함

- healthcheck에 포트 80 listening 확인 로직까지 추가함



## 오늘 배운 점

- 명령 결과는 `$(...)`로 변수에 저장해서 조건문에 사용할 수 있다.

- `\[ ]`, `\[\[ ]]`, `sh`, `bash`의 차이를 직접 확인했다.

- `grep -q`는 출력 없이 성공/실패만 확인할 수 있어서 healthcheck에 잘 맞는다.

- HTTP 상태는 문자열 `OK`보다 상태코드 `200` 기준으로 보는 쪽이 더 실용적이라는 점을 이해했다.



## 결과/증거

- 기존 `baseline-check.md` 기준을 바탕으로 healthcheck 작성 방향을 정리함

- `healthcheck.sh` 작성 완료

- nginx 상태, localhost 응답, port 80 listening 확인 로직 반영

- 스크립트 실행 결과 확인



## 막힌 점

- `curl -I` 결과를 `awk`로 비교할 때 예상과 다르게 동작하는 부분이 있었다.

- `\[\[ ]]` 문법을 사용한 스크립트를 `sh`로 실행해서 오류가 발생했다.

- baseline check 전체 자동화와 healthcheck 범위를 어디까지 둘지 고민했지만 우선 핵심 항목만 넣는 것으로 정리했다.
