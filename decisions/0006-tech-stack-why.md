## Context

Day 1~24까지 진행하면서 nginx, Docker, Reverse Proxy, Docker Compose를 사용했다.
당시에는 "왜 이 기술을 선택했는가"를 명시적으로 정리하지 않았기 때문에

## 기술별 선택 근거

### 1. nginx

| 항목 | 내용 |
|------|------|
| 선택한 것 | nginx |
| 대안 | Apache, Caddy |
| 선택 이유 | 설정 문법이 단순하고 실무에서 가장 많이 쓰인다 |

**왜 썼나:**
보안과 역할 분리를 위해 사용했다.
클라이언트 요청을 nginx가 앞단에서 받아서 라우팅, 로깅, 보안 헤더 처리를 담당하고
뒤쪽 앱은 실제 응답만 처리하도록 역할을 나눴다.

**대안과의 비교:**
Apache도 같은 역할을 할 수 있지만 설정 파일 구조가 복잡하다.
Caddy는 자동 HTTPS 기능이 있지만 학습 자료가 적다.
nginx는 설정이 단순하고 실무에서 가장 많이 쓰이기 때문에 선택했다.

**면접 한 줄 설명:**
"nginx를 선택한 이유는 설정이 단순하고 실무에서 많이 쓰이기 때문입니다.
Apache도 대안이었지만 설정 복잡도를 고려해 nginx를 선택했습니다.
실제로 upstream 포트를 잘못 설정해서 502가 발생했고, error.log로 원인을 찾아 복구했습니다."

---

### 2. Docker

| 항목 | 내용 |
|------|------|
| 선택한 것 | Docker 컨테이너 |
| 대안 | apt로 직접 설치 |
| 선택 이유 | 환경 격리로 충돌 방지 + 재현 가능성 |

**왜 썼나:**
앱을 EC2에 직접 설치하면 호스트 환경과 충돌이 발생할 수 있다.
Docker는 컨테이너 안에 별도의 Linux 환경을 띄워서 호스트에 영향을 주지 않고
독립적으로 실행할 수 있다.

**대안과의 비교:**
apt로 직접 설치하면 환경 오염 위험이 있고 다른 환경에서 재현하기 어렵다.
Docker는 격리된 환경을 제공하고 `docker run` 한 줄로 재현 가능하다.

**면접 한 줄 설명:**
"Docker를 선택한 이유는 환경 격리로 충돌을 방지하고 재현 가능한 구조를 만들기 위해서입니다.
실제로 포트 매핑 오타(8080:8080 → 8080:80)로 접속 실패를 재현하고 복구했습니다."

---

### 3. Reverse Proxy

| 항목 | 내용 |
|------|------|
| 선택한 것 | nginx Reverse Proxy |
| 대안 | 클라이언트가 앱 포트에 직접 접근 |
| 선택 이유 | 외부에 열리는 포트를 80 하나로 최소화하여 공격 표면 축소 |

**왜 썼나:**
클라이언트가 Docker 앱(8080)에 직접 접근하려면 UFW에서 8080을 열어줘야 한다.
앱이 늘어날수록 열어야 하는 포트도 늘어나고 공격 표면이 넓어진다.
nginx를 중간에 두면 80포트 하나만 열어두고 뒤쪽 앱 포트는 외부에 숨길 수 있다.

**대안과의 비교:**
앱 포트를 직접 열면 구조는 단순하지만 보안상 위험하다.
Reverse Proxy를 쓰면 포트 노출을 최소화하고 nginx에서 라우팅을 일괄 관리할 수 있다.

**면접 한 줄 설명:**
"Reverse Proxy를 선택한 이유는 외부에 열리는 포트를 80 하나로 줄여 공격 표면을 최소화하기 위해서입니다.
실제로 proxy_pass 포트 오타로 502가 발생했고, error.log에서 upstream 연결 실패를 확인하고 복구했습니다."

---

### 4. Docker Compose

| 항목 | 내용 |
|------|------|
| 선택한 것 | Docker Compose |
| 대안 | docker run 명령어로 직접 실행 |
| 선택 이유 | 파일로 관리 → GitHub 업로드 가능 + IaC의 시작 |

**왜 썼나:**
`docker run`은 터미널 명령어라 실행 후 사라지고 GitHub에 올릴 수 없다.
`docker-compose.yml`은 파일이기 때문에 GitHub에 올릴 수 있고
다른 환경에서도 똑같이 재현할 수 있다.
이게 인프라를 코드로 관리하는 IaC 개념의 시작이다.

**대안과의 비교:**
`docker run`은 명령이 길어지고 실수 가능성이 높으며 버전 관리가 안 된다.
Compose는 선언형으로 관리되어 변경 이력이 남고 재현이 쉽다.

**면접 한 줄 설명:**
"Docker Compose를 선택한 이유는 인프라를 파일로 관리해서 GitHub에 올리고 재현 가능한 구조를 만들기 위해서입니다.
실제로 compose 파일 포트 오타로 접속 실패를 재현하고 --force-recreate로 복구했습니다."

## 정리 한눈에 보기

| 기술 | 왜 썼나 | 대안 | 선택 이유 |
|------|---------|------|-----------|
| nginx | 보안 + 역할 분리 | Apache, Caddy | 설정 단순 + 실무 표준 |
| Docker | 환경 격리 | apt 직접 설치 | 충돌 방지 + 재현 가능성 |
| Reverse Proxy | 포트 노출 최소화 | 앱 직접 접근 | 공격 표면 축소 |
| Docker Compose | 재현 가능성 | docker run | 파일 관리 + IaC 시작 |
